webpack高级

  • webpack高级主要是对webpack进行优化，在编译/运行代码时有更好的性能
  • 主要有四个方面
    • 提升开发体验
    • 提升打包构建速度
    • 减少代码体积
    • 优化代码运行性能

提升开发体验

SourceMap(源代码映射)

场景

  • 在打包的过程中会将所有css代码合并为一个文件(提取css文件)，所有js代码合并为一个文件，并且会自动添加一些注释信息。
  • 运行代码过程中如果出错，日志标识的是打包构建之后的文件报错的位置
  • 难以找到报错的源代码，不利于开发维护

方案

  • 使用webpack提供的开发工具devtool
  • 在webpack配置文件中配置devtool工具
  • 原理为SourceMap(源代码映射)，用来生成源代码与构建后代码一一映射的文件的方案
  • 编译时自动在同级目录下生成一个源文件名.map文件，里面包含源代码和构建后代码每一行、每一列的映射关系。当构建后代码出错了，会通过该.map 文件映射到源代码报错位置，从而让浏览器提示源代码文件出错位置
  • 具体取值可查阅webpack devtool文档，以下为两种常用取值

使用

开发模式
  • 使用cheap-module-source-map的SourceMap映射方案
  • 只包含行映射，不包含列映射，打包编译速度相对快
module.exports = {
    /*
        其他配置项
    */
    mode: "development",
    devtool: "cheap-module-source-map", // 只包含行映射，不包含列映射
};
生产模式
  • 使用source-map的SourceMap映射方案
  • 既包含行映射，又包含列映射，打包编译速度相对慢
module.exports = {
    /*
        其他配置项
    */
    mode: "production",
    devtool: "source-map", // 既包含行映射，又包含列映射
};

提升打包构建速度

HMR(热模块替换)

场景

  • 当我们修改代码时，webpack 默认会将所有模块全部重新打包编译，速度很慢

方案

  • 在webpack开发服务器中，启动热模块替换(HMR)功能
  • 修改某个模块代码时，只有这个模块代码需要重新打包编译，其他模块不变
  • 只有在开发模式，且以开发服务器方式启动时有效

使用

  • 在开发服务器的配置中，hot属性即是否开启HMR，默认值即为true
  • 样式loader中已经对样式进行了处理，只要开启了HMR即可
  • js代码需要使用module.hot.accept对每个js资源进行监听，但是在项目中可使用js也进行处理过的loader，如(vue-loader, react-hot-loader

./config/webpack.dev.js

module.exports = {
    /*
        其他配置项
    */
    mode: "development",
    devServer: {
        host: "localhost", // 启动服务器域名
        port: "3000", // 启动服务器端口号
        open: true, // 是否自动打开浏览器
        hot: true, // 是否开启HMR
    },
};

./src/main.js

import count from './js/count';
import sum from './js/sum';

import './css/index.css';
import './css/index.less';
import './css/index.sass';
import './css/index.scss';
import './css/index.styl';

import './css/iconfont.css';

console.log(count(3, 2));
console.log(sum(1, 2, 3, 4));

// 判断是否支持HMR功能
if (module.hot) {
    // 给相应模块添加HMR
    // module.hot.accept('./js/count', () => console.log('count.js changed'));
    // module.hot.accept('./js/sum', () => console.log('sum.js changed'));
    module.hot.accept(['./js/count', './js/sum'], () => console.log('count.js or sum.js changed'));
}

OneOf(rules规则匹配到一项就退出)

场景

  • webpack默认打包过程中，对每个文件，都会将所有rules规则匹配一次
  • 实际上某些rule规则，匹配是互斥关系，即对于一个文件，只会在这些rule中匹配一项进行处理

方案

  • 对于那些具有互斥关系的rule规则，使用oneOf包裹
  • 对于oneOf包裹的rules规则，只要匹配到了一项rule，则后面的rules则会忽略跳过

使用

module.exports = {
    /*
        其他配置项
    */
    // 加载器
    module: {
        rules: [
            // 加载器配置
            {
                // 只选择一项rule，后面的忽略退出
                oneOf: [
                    // 加载器配置
                    {
                        test: /\.css$/,  // 正则匹配文件名，只检测匹配到的文件
                        use: getStyleLoaders(),
                    },
                    {
                        test: /\.less$/,
                        use: getStyleLoaders('less-loader'),
                    },
                    {
                        test: /\.s[ac]ss$/,
                        use: getStyleLoaders('sass-loader'),
                    },
                    {
                        test: /\.styl$/,
                        use: getStyleLoaders('stylus-loader'),
                    },

                    {
                        test: /\.(ttf|woff2?|mp3|mp4|avi|mkv)$/,
                        type: "asset/resource",
                        generator: {
                            // 匹配到的文件输出路径
                            // 将文件输出到 static/media 目录中
                            // 将图片文件命名 [hash:10][ext][query]
                            // [hash:10]: hash值取10位
                            // [ext]: 使用之前的文件扩展名
                            // [query]: 添加之前的query参数
                            filename: 'static/media/[hash:10][ext][query]',
                        }
                    },
                    {
                        test: /\.svg$/,
                        type: "asset/inline",
                    },
                    {
                        test: /\.(png|jpe?g|gif|bmp)$/,
                        type: "asset",
                        parser: {
                            dataUrlCondition: {
                                maxSize: 10 * 1024 // 小于10kb的图片会被base64处理,即data URI
                            }
                        },
                        generator: {
                            // 匹配到的文件输出路径
                            // 将文件输出到 static/images 目录中
                            // 将图片文件命名 [hash:10][ext][query]
                            // [hash:10]: hash值取10位
                            // [ext]: 使用之前的文件扩展名
                            // [query]: 添加之前的query参数
                            filename: 'static/images/[hash:10][ext][query]',
                        },
                    },
                    {
                        test: /\.txt$/,
                        type: "asset/resource",
                    },

                    {
                        test: /\.js$/,
                        exclude: /node_modules/, // 排除node_modules代码不编译
                        loader: "babel-loader",
                    },
                ]
            },
            // 其他rules规则
            {
                
            }
        ],
    },
};

Include(只处理该目录) / Exclude(不处理该目录)

场景

  • 开发时我们会使用到多种三方依赖包和插件，而这些文件是不需要经过webpack编译可以直接使用的
  • 我们希望在编译打包时，可以排除这些文件，或者只打包我们需要处理的文件

方案

  • 在rules规则和plugins插件配置中，使用Include / Exclude，来使loader和plugin处理相应的目录
  • include代表只处理该目录或者该文件，exclude代表不处理该目录或该文件(其他文件均需处理)

使用

module.exports = {
    /*
        其他配置项
    */
    // 加载器
    module: {
        rules: [
            // 加载器配置
            {
                // 只选择一项rule，后面的忽略退出
                oneOf: [
                    // 加载器配置
                    /*
                        其他rule
                    */
                    {
                        test: /\.js$/,
                        // exclude: /node_modules/, // 排除node_modules代码不编译
                        include: /src/, // 只处理src目录下的文件
                        loader: "babel-loader",
                    },
                ]
            },
            // 其他rules规则
            {
                
            }
        ],
        
    },
    // 插件
    plugins: [
        // 插件配置
        
        /*
            其他插件配置
        */

        // 使用ESLint进行js代码格式检查
        new ESLintWebpackPlugin({
            // 指定检查文件的根目录
            context: path.resolve(__dirname, "../src"),
            exclude: 'node_module', // 默认值, 此处只能是字符串，不能是正则
        }),
    ],
};

Cache(ESLint和Babel缓存)

场景

  • 每次打包时eslint和babel都需要对每个js文件进行检查和编译

方案

  • 开启eslint和babel缓存
  • 每次打包时，读取上次js文件检查和编译结果的缓存，并缓存这次检查和编译结果的结果

使用

module.exports = {
    /*
        其他配置项
    */
    // 加载器
    module: {
        rules: [
            // 加载器配置
            {
                // 只选择一项rule，后面的忽略退出
                oneOf: [
                    // 加载器配置
                    /*
                        其他rule
                    */
                    {
                        test: /\.js$/,
                        // exclude: /node_modules/, // 排除node_modules代码不编译
                        include: /src/, // 只处理src目录下的文件
                        loader: "babel-loader",
                        options: {
                            // 缓存路径为node_modules/.cache/babel-loader
                            cacheDirectory: true,    // 是否开启babel编译缓存
                            cacheCompression: false, // 是否压缩缓存文件
                        }
                    },
                ]
            },
            // 其他rules规则
            {
                
            }
        ],
        
    },
    // 插件
    plugins: [
        // 插件配置
        
        /*
            其他插件配置
        */

        // 使用ESLint进行js代码格式检查
        new ESLintWebpackPlugin({
            // 指定检查文件的根目录
            context: path.resolve(__dirname, "../src"),
            exclude: /node_module/, // 默认值
            cache: true, // 开启缓存
            cacheLocation: path.resolve(__dirname, '../node_modules/.cache/.eslintcache'),  // 缓存路径
        }),
    ],
};

Thread(多进程打包)

场景

  • 当项目体积庞大时，打包速度将会变得很慢，而且其中众多是js文件
  • 压缩(图片等资源)等工作也变得十分耗时

方案

  • 使用thread-loader对js文件的检查、编译打包和文件的压缩工作进行多线程构建打包

  • 获取CPU的核数，即启动多进程的最大进程数

  • 仅在特别耗时的操作中使用，因为每个进程启动就有大约为 600ms 左右开销

使用

/*
    其他引入
*/

// 引入压缩插件
const TerserPlugin = require("terser-webpack-plugin");

const os = require('os');  // 引入os模块
const threads = os.cpus().length - 1; // 获取cpu核数,预留单核供其他使用

module.exports = {
    /*
        其他配置项
    */
    // 加载器
    module: {
        rules: [
            // 加载器配置
            {
                // 只选择一项rule，后面的忽略退出
                oneOf: [
                    // 加载器配置
                    /*
                        其他rule
                    */
                    {
                        test: /\.js$/,
                        // exclude: /node_modules/, // 排除node_modules代码不编译
                        include: /src/, // 只处理src目录下的文件
                        use: [
                            {
                                loader: 'thread-loader', // 使用多进程打包
                                options: {
                                    workers: threads, // 开启的进程数量
                                },
                            },
                            {
                                loader: 'babel-loader', // 使用babel进行兼容性处理
                                options: {
                                    cacheDirectory: true,    // 是否开启babel编译缓存
                                    cacheCompression: false, // 是否压缩缓存文件
                                }
                            },
                        ],
                    },
                ]
            },
            // 其他rules规则
            {
                
            }
        ],
        
    },
    // 插件
    plugins: [
        // 插件配置
        
        /*
            其他插件配置
        */

        // 使用ESLint进行js代码格式检查
        new ESLintWebpackPlugin({
            // 指定检查文件的根目录
            context: path.resolve(__dirname, "../src"),
            exclude: /node_module/, // 默认值
            cache: true, // 开启缓存
            cacheLocation: path.resolve(__dirname, '../node_modules/.cache/.eslintcache'),  // 缓存路径
            threads, // 开启的进程数量，默认1(即不开启多进程)
        }),
    ],
    // 优化
    optimization: {
        minimize: true, // 是否开启js代码压缩，生产模式默认开启
        minimizer: [
            // 此处配置压缩插件，与写在plugins等价，但推荐写在这儿
            // css压缩
            new CssMinimizerPlugin(),
            // 当生产模式会默认开启TerserPlugin，但是我们需要进行其他配置，就要重新写了
            new TerserPlugin({
                parallel: threads // 开启进程数量
            })
        ],
    },
};

减少代码体积

Tree Shaking(移除未使用的代码)

场景

  • 当引入三方依赖包、工具库、组件库时，我们或许只会使用到其中的极小部分功能

方案

  • 使用Tree Shaking技术，用于移除没有使用上的代码，依赖于ES Module

使用

  • Webpack 已经默认开启了这个功能，无需其他配置，但需要清楚该项技术

Babel Runtime(Babel辅助代码的模块库)

场景

  • babel在做js代码兼容性处理时，会在每个js文件生成大量的辅助代码
  • 每个js文件的辅助代码之间，有极大的重复性

方案

  • Babel Runtime就是这些辅助代码的集合库，借此，可以让每个js文件不生成辅助代码，进而替代为从集合库中引入所需的辅助代码，从而减少代码体积，可参阅Babel配置

  • @babel/plugin-transform-runtime就可以实现该功能，禁止babel对每个js文件生成辅助代码，替代为从中引入所需的辅助代码

  • 该插件为babel插件，需要在babel-loader中进行配置

使用

module.exports = {
    /*
        其他配置项
    */
    // 加载器
    module: {
        rules: [
            // 加载器配置
            {
                // 只选择一项rule，后面的忽略退出
                oneOf: [
                    // 加载器配置
                    /*
                        其他rule
                    */
                    {
                            test: /\.js$/,
                            // exclude: /node_modules/, // 排除node_modules代码不编译
                            include: /src/, // 只处理src目录下的文件
                            use: [
                                {
                                    loader: 'thread-loader', // 使用多进程打包
                                    options: {
                                        workers: threads, // 开启的进程数量
                                    },
                                },
                                {
                                    loader: 'babel-loader', // 使用babel进行兼容性处理
                                    options: {
                                        cacheDirectory: true,    // 是否开启babel编译缓存
                                        cacheCompression: false, // 是否压缩缓存文件
                                        plugins: ['@babel/plugin-transform-runtime'], // 禁止辅助代码生成，改为引入
                                    }
                                },
                            ],
                        },
                ]
            },
            // 其他rules规则
            {
                
            }
        ],
        
    },
};

Image Minimizer(图片压缩)

场景

  • 当项目中引入了大量图片时，图片会占用很大的体积，请求速度变慢

方案

  • 使用image-minimizer-webpack-plugin插件来压缩图片

  • 包含有有损压缩和无损压缩两种方式

    • 有损压缩需下载npm install imagemin-gifsicle imagemin-mozjpeg imagemin-pngquant imagemin-svgo -D
    • 无损压缩需下载npm install imagemin-gifsicle imagemin-jpegtran imagemin-optipng imagemin-svgo -D
    • gifsicle下载失败时，是因为raw.githubusercontent.com遭受污染，可查出该域名目前对应的ip地址，配置hosts再重新下载

  • 以下为无损压缩为例，如果没有将所需jpegtran.exe和optipng.exe软件下载到node_module中，可查阅JPEGtran 官网地址和optiPNG 官网地址下载，手动添加到node_module相应目录，有损压缩则为mozJPEG 和PNGquant

使用

module.exports = {
    /*
        其他配置项
    */
    // 优化
    optimization: {
        minimize: true, // 是否开启js代码压缩，生产模式默认开启
        minimizer: [
            // 此处配置压缩插件，与写在plugins等价，但推荐写在这儿
            // css压缩
            new CssMinimizerPlugin(),
            // 当生产模式会默认开启TerserPlugin，但是我们需要进行其他配置，就要重新写了
            new TerserPlugin({
                parallel: threads // 开启进程数量
            }),
            
            // 压缩图片
            new ImageMinimizerPlugin({
                minimizer: {
                    implementation: ImageMinimizerPlugin.imageminGenerate,
                    options: {
                        plugins: [
                            ["gifsicle", { interlaced: true }],
                            ["jpegtran", { progressive: true }],
                            ["optipng", { optimizationLevel: 5 }],
                            ["svgo",{
                                plugins: [
                                    "preset-default",
                                    "prefixIds",
                                    {
                                        name: "sortAttrs",
                                        params: {
                                            xmlnsOrder: "alphabetical",
                                        },
                                    },
                                ],
                            }],
                        ],
                    },
                },
            }),
        ],
    },
};

优化代码运行性能

Code Split(代码分割)

详见webpack优化之代码分割

Preload / Prefetch(预加载)

场景

  • 在Code Split(代码分割)中我们采用了按需加载的方式来提升首屏渲染性能
  • 在此情况下，当用户访问被按需加载处理过的页面或功能时，如果资源体积较大，那么用户会感觉到明显卡顿效果

方案

  • 首先加载当前页面所需资源，按需加载资源暂不加载
  • 当前页面所需资源加载完成后，再加载后续需要使用的资源
  • 使用@vue/preload-webpack-plugin插件，并进行相关配置
  • 有Preload / Prefetch两种预加载方式，它们都只会加载资源并缓存，并不执行
    • Preload：告诉浏览器立即加载资源，只能加载当前页面需要使用的资源，加载优先级高，兼容性较好
    • Prefetch：告诉浏览器在空闲时才开始加载资源，可以加载当前页面资源，也可以加载下一个页面需要使用的资源，兼容性较差

使用

module.exports = {
    /*
        其他配置项
    */
    
    // 插件
    plugins: [
        // 插件配置
        
        /*
            其他插件配置
        */

        // 预加载
        new PreloadWebpackPlugin({
            rel: 'preload', // 有preload / prefetch两种取值，在prefetch下不可设置as
            as: 'script', // 作为script标签进行加载
        })
    ],
};

Network Cache

场景.1

  • 在开发中通常会对静态资源进行缓存优化，浏览器第二次请求资源就能读取缓存，速度更快
  • 如果修改代码前后输出的文件名是一样的，将来发布新版本，因为文件名没有变化导致浏览器会直接读取缓存，不会加载新资源，项目也就没法更新

方案.1

  • 可以在文件名上添加哈希值来达到文件名始终一样的场景，常见的有三种哈希值，这里我们使用contenthash
    • fullhash(webpack4 是 hash): 每次修改任何一个文件，所有文件名的 hash 至都将改变。所以一旦修改了任何一个文件，整个项目的文件缓存都将失效。
    • chunkhash: 根据不同的入口文件(Entry)进行依赖文件解析、构建对应的 chunk，生成对应的哈希值。
    • contenthash: 根据文件内容生成 hash 值，只有文件内容变化了，hash 值才会变化，并且只会修改该文件的hash值，每个文件 的hash 值都是唯一不重复的。

场景.2

  • 使用contenthash时，会根据文件内容的变化来改变该文件的hash值，
  • 如果有其他文件引入了该文件时，其他文件没有进行修改
  • 打包时由于引入路径中的文件名发生了变化，也会认为文件内容发生了改变，进而导致其他文件hash值改变，缓存失效

方案.2

  • 在optimization中配置runtimeChunk，来避免文件发生改变时，进入该文件的文件由于引入路径的关系跟着改变
  • 将文件的hash值交给runtimeChunk, 来保存文件的 hash 值、保存的hash值与文件的关系，整个文件体积较小，变化导致重新请求的代价也较小。

使用

module.exports = {
    /*
        其他配置项
    */
    // 输出
    output: {
        // 所有文件输出路径
        path: path.resolve(__dirname, '../dist'),  // 绝对路径
        // 入口文件输出文件名, 该值同样是splitChunks的文件输出命名格式
        // [name]是webpack命名规则，使用chunk的name作为输出bundle的文件名。
        // module、chunk、bundle是什么？打包前的资源是module, 打包完成资源是chunk，最后输出的是bundle
        // 这里的name是entry中的键名，和文件名无关，即front和manage。
        // contenthash是据文件内容生成 hash 值，随着内容的改变而改变
        // filename: "main.js",
        filename: "static/js/[name].[contenthash:8].js",
        chunkFilename: "static/js/chunk/[name].[contenthash:8].chunk.js", // 此处的name可读取魔法命名的值
        assetModuleFilename: "static/media/[hash:8].[ext][query]",
        // 禁止箭头函数
        environment: {
            arrowFunction: false,
        },
        clean: true,
    },
    // 优化
    optimization: {
        /*
            其他优化配置
        */

        // 提取runtime文件, 来管理hash值与文件的关系
        runtimeChunk: {
            name: (entrypoint) => `runtime~${entrypoint.name}`, // runtime文件命名规则, 最终作为name值在output中的filename下命名
        },
    },
};

Core-js

场景

  • babel的智能预设@babel/preset-env能处理一般情况下的es6兼容性问题
  • 如果使用es6以上语法或者是智能预设不处理的语法时，如 async /await、Promise等等，将仍然存在兼容性问题

方案

  • 使用 core-js，来处理es6及以上的语法特性

  • core-js 是专门用来做 ES6 以及以上 API 的 polyfill, 即补丁

  • 可以直接在代码中导入整个补丁，也可按需加载我们使用的补丁，但都比较麻烦

  • 可以在babel配置文件中进行配置，根据我们代码中使用的语法，来自动按需加载相应的 polyfill 了

  • 可能新语法eslint会检查报错，如使用Promise时，需在eslint中配置es6全局环境

使用

./eslintrc.js

module.exports = {
    // 继承 Eslint 官方规则
    extends: ["eslint:recommended"],
    // 设置环境
    env: {
        node: true, // 启用node中全局变量
        browser: true, // 启用浏览器中全局变量
        es6: true, // 启用es6中全局变量
    },
    // 设置解析选项
    parserOptions: {
        ecmaVersion: 11, // 设置ES语法版本, es11才有动态导入语法
        sourceType: "module", // 设置模块化语法
        allowImportExportEverywhere: true,
    },
    rules: {
        "no-var": 2, // 不能使用 var 定义变量 2--‘error’
    },
};

./babel.config.js

module.exports = {
    presets: [
        [
            "@babel/preset-env", // 智能预设
            // 按需加载core-js的polyfill
            { useBuiltIns: "usage", corejs: { version: "3", proposals: true } },
        ]
    ],
};

PWA(渐进式网络应用程序)

场景

  • 开发 Web App 项目时，项目一旦处于网络离线情况，就无法访问了
  • 希望能够给项目提供离线体验

方案

  • 使用workbox-webpack-plugin插件来配置注册Service Worker，来实现PWA(渐进式网络应用程序)
  • workbox-webpack-plugin插件需在webpack配置文件中进行配置
  • 在所有入口(如果是多入口)文件下注册 Service Worker
  • 需部署在服务器上才可看到效果，如开发服务器

使用

./config/webpack.dev.js

module.exports = {
    /*
        其他配置项
    */
    
    // 插件
    plugins: [
        // 插件配置
        
        /*
            其他插件配置
        */

        // 配置ServiceWorkers
        new WorkboxPlugin.GenerateSW({
            // 这些选项帮助快速启用 ServiceWorkers
            // 不允许遗留任何“旧的” ServiceWorkers
            clientsClaim: true,
            skipWaiting: true,
        })
    ],
    // 模式
    mode: "development",
    // 开发服务器
    devServer: {
        host: "localhost", // 启动服务器域名
        port: "3000", // 启动服务器端口号
        open: true, // 是否自动打开浏览器
        hot: true, // 是否开启HMR
    },
};

./src/main.js

import { sum, count } from './js/math';

console.log(count(3, 2));
console.log(sum(1, 2, 3, 4));

let timer = setTimeout(() => {
    // eslint会对动态导入语法报错，需要修改eslint配置文件
    // 下面的 `/* webpackChunkName: "tip" */`是webpack给动态导入模块chunk命名的方式, 为webpack魔法命名
    // "tip"将来就会作为[name]的值显示。
    import(/* webpackChunkName: "tip" */'./js/tip').then((m) => {
        m.default();
        clearTimeout(timer);
        timer = null;
    })
}, 3000);

// 判断是否支持HMR功能
if (module.hot) {
    // 给相应模块添加HMR
    // module.hot.accept('./js/count', () => console.log('count.js changed'));
    // module.hot.accept('./js/sum', () => console.log('sum.js changed'));
    module.hot.accept(['./js/math', './js/tip'], () => console.log('math.js or tip.js changed'));
}

// 注册 Service Worker
if ('serviceWorker' in navigator) {
    window.addEventListener('load', () => {
        // 此处应为相对路径
        navigator.serviceWorker.register('service-worker.js').then(registration => {
            console.log('SW registered: ', registration);
        }).catch(registrationError => {
            console.log('SW registration failed: ', registrationError);
        });
    });
}
